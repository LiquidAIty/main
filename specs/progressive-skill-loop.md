# Progressive Skill Loop Spec

> Transition policy: this is a legacy/source document. Reusable learning still belongs in skills,
> but ordinary jobs now use one temporary CoderPacket/spec-as-prompt rather than durable specs.

## Purpose

Stop progressive work from breaking unrelated systems by making skills the durable memory of
repo-specific attempts, successes, failures, guardrails, proofs, and query-ready examples.

## Lifecycle

1. User prompt arrives.
2. Planner or Codex reads `PLAN.md`.
3. Planner or Codex finds the relevant spec.
4. Planner or Codex refreshes CBM.
5. Planner or Codex searches existing skills.
6. If a matching skill exists, append an attempt block to that skill.
7. If no matching skill exists, create a short skill stub and append the attempt there.
8. Fable executes the bounded attempt.
9. Fable writes result, proof, and CBM-after evidence into the attempt.
10. Success updates Current Procedure and Successful Examples.
11. Failure updates Failed Attempts and Guardrails.
12. Future agents query skills before touching related code.

## Requirements

* One skill equals one graphable Markdown file by default.
* Skill files live at `skills/<skillname>.md`.
* No skill subfolders unless explicitly requested.
* No separate JSON, example, or query files unless generated by a later importer or exporter.
* Skill files may contain graphable Markdown or OWL-ish metadata.
* Markdown is the authoring format.
* A later importer may convert graphable lines into JSON, Postgres, or ThinkGraph.
* Code snippets are retrieved fresh by query.
* Raw diffs are not durable memory.
* Every real code attempt must attach to a skill.
* Successful code attempts must update or create skill memory.
* Failed attempts must update a skill with guardrail and retry metadata.
* Fresh CBM is required before creating an attempt and after code changes.

## Non-Goals

This spec does not create or execute the first real attempt. It does not start Fable, execute T001,
or implement runtime behavior.

## Validation

* `AGENTS.md` defines the progressive skill loop as repo law.
* `PLAN.md` defines PLAN, specs, skills, and CodeGraph as the future GraphRAG planning context.
* No attempt block is created from the process-normalization prompt.
* The next real user task can become the first real skill attempt.

## Risks

* Unbounded attempts can turn skills into task logs instead of reusable memory.
* Stale CBM can attach attempts to incorrect files or symbols.
* Recording process-only prompts as attempts would pollute the skill graph.
